## Grep Drill

Các lệnh kiểm tra log cần chạy khi app ghi log ra file:

```bash
rg '"correlation_id":"err-id-123"' /tmp/day12-api.log
rg '"level":"(warning|error|critical)"' /tmp/day12-api.log
rg '"event":"login_failed".*"reason":"invalid_credentials"' /tmp/day12-api.log
```

Ý nghĩa:

- Lệnh 1 gom toàn bộ log của một request cụ thể.
- Lệnh 2 tìm các log từ `WARNING` trở lên để ưu tiên điều tra.
- Lệnh 3 lọc login failure theo event và reason, không cần grep email hoặc password.

## Metrics Sketch

- `http_requests_total` counter, labels `method`, `path`, `status`; alert khi tỉ lệ `5xx` lớn hơn 5% trong 5 phút.
- `http_request_duration_seconds` histogram, labels `method`, `path`; alert khi p95 của endpoint chính lớn hơn 500ms trong 10 phút.
- `auth_login_failures_total` counter, labels `reason`, `email_domain`; alert khi một domain hoặc toàn hệ thống tăng bất thường trong 5 phút.
- `db_query_duration_seconds` histogram, labels `operation`, `table`; alert khi p95 query đọc/ghi vượt 300ms trong 10 phút.
- `unhandled_exceptions_total` counter, labels `exception_type`, `path`; alert ngay khi có hơn 3 lỗi mới trong 5 phút.

  - cold, xóa key trước mỗi request: p50 `3.497ms`, p99 `6.257ms`.
  - warm, cache đã populate: p50 `2.854ms`, p99 `4.143ms`.

Failure modes:

- Stale data: nếu update post nhưng quên `DEL v1:post:{id}`, user có thể thấy dữ liệu cũ tối đa 5 phút. Cách phát hiện là so sánh DB và response API sau write; cách giảm là test invalidation cho mọi write path.
- Cache stampede: nếu một key nóng hết TTL, nhiều request cùng miss và dồn vào DB. Với bài này traffic nhỏ nên chưa xử lý; production có thể dùng lock ngắn hạn hoặc TTL jitter.
- Redis outage: nếu Redis down, endpoint cached hiện có thể fail thay vì fallback DB. Production nên catch Redis errors, log warning và đọc DB trực tiếp.
