## Objective

Stand up the project skeleton вЂ” config, layout, DB models, migrations, Docker, CI вЂ” so that on Day 18 you can focus entirely on writing route handlers and business logic without architectural decisions blocking you.

## Why this matters

A clean scaffold pays for itself every hour afterward. Today is mostly mechanical work translating Weeks 1вЂ“3 patterns into the new project, but the discipline matters: every shortcut you take today is a debt you'll service on Day 19 when you're racing the clock.

## What to copy vs. what to rewrite

You can lift wholesale from your Week 1вЂ“3 work:
- `Dockerfile` + `.dockerignore` (Day 13).
- `docker-compose.yml` skeleton (Day 13).
- `app/config.py` (Day 5).
- `app/logging_setup.py` and `CorrelationIdMiddleware` (Day 12).
- `app/db.py` and `get_session` dependency (Day 8).
- Test fixtures: `conftest.py` engine/session/client (Day 11).
- `terraform/` directory (Day 15).
- `.github/workflows/ci.yml` (Day 15).
- `pyproject.toml` deps section.

You should *rewrite* (or write fresh):
- Models for your domain.
- Migrations (the initial one is autogenerated from your new models).
- Routes and services вЂ” these are specific to your design.
- Pydantic schemas.
- The README.

## Day plan (suggested)

**Morning (3 hours)**

1. Repo scaffolding вЂ” `uv init`, deps, layout above.
2. Wire up: `config.py`, `db.py`, `logging_setup.py`, `middleware.py`.
3. `create_app()` factory in `main.py` that registers middleware, routes, exception handlers.
4. Stub out routes вЂ” every endpoint from your design doc, returning `501 Not Implemented` for now. Confirm `/docs` lists them all.

**Afternoon (5 hours)**

5. Define all SQLAlchemy models. Use the typed style. Add proper relationships and cascade rules.
6. Initialize Alembic. Generate the first migration. **Read it.** Apply it.
7. `docker-compose.yml` with api + db + cache. Verify `docker compose up -d` brings everything up and `/docs` is reachable on `http://localhost:8000/docs`.
8. CI workflow вЂ” lint, test, docker build. Push and confirm green.
9. Health endpoints вЂ” `/health` (always 200) and `/ready` (503 if DB unreachable). Tests for both.
10. Commit and push frequently. Open a draft PR titled `feat: scaffold` so the mentor can comment as you work.

## Exercises (the deliverable)

This is execution day. The "exercise" is the scaffold itself. Concretely:

1. **Repo + structure** вЂ” match the layout above. Empty modules are fine; just create the skeleton.
2. **Models + first migration** вЂ” `alembic revision --autogenerate` produces something faithful to your design's ER diagram. `alembic upgrade head` succeeds from an empty DB.
3. **Compose stack** вЂ” `docker compose up -d` then `curl http://localhost:8000/health` returns 200.
4. **Endpoints listed in `/docs`** вЂ” every design-doc endpoint is registered (even if returning 501).
5. **CI** вЂ” green on the first PR. Lint, tests (you'll only have a few вЂ” the health-endpoint tests are enough today), and `docker build`.
6. **README** stub вЂ” install / run / test commands. Fill it out as you go this week.

## Common pitfalls

- **Skipping the `services/` layer** вЂ” "I'll just put it in the route" is fine today and painful by Friday. Set up the layer now even if it feels heavy.
- **Mixing autogenerated migrations with manual edits without reading them** вЂ” happens fast on Day 17. Read every migration line.
- **`docker compose up` works but tests fail** вЂ” usually a mismatch between the dev DB and the test DB URL. Set both in `.env.example`.
- **No `.env.example`** вЂ” the mentor can't run your project on Day 20 without one.
- **Committing `.env` with real secrets** вЂ” set this up early; `.env` should be gitignored from commit 1.
- **Importing models from inside `app/db.py` causes circular imports** вЂ” keep imports in one direction: `models в†’ db`, never the reverse.

## Self-check

1. A different engineer clones the repo at end of day. What's the *minimum* sequence of commands to get tests passing? Is it documented in the README?
2. Where is each of the following set: log level, DB URL, JWT secret, CORS origins? (Spoiler: should all be in `config.py`/`Settings` with env defaults.)
3. Why are routes stubbed with 501 rather than just absent?
4. Your initial migration produced more (or fewer) tables than your ER diagram. Now what?
5. CI passes locally with `pytest`. CI fails in GitHub. What are the two most likely causes?
6. The compose `api` service exits immediately. How do you debug it?
